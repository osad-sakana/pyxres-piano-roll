# Pyxel互換・音楽エディタ 設計書

[[260708_Pyxel互換音楽エディタ実現可能性調査]]の結論（方式B：Pyxel非依存の自前書き出しが現実的）を受けた設計書。実現可能性調査で確認済みの事実は再掲せず、設計判断とその根拠のみを記述する。

## 1. 基本方針

| 項目 | 決定 | 根拠 |
| --- | --- | --- |
| プラットフォーム | ブラウザ単体（単一HTMLファイル） | 配布・起動が最も容易。ビルド環境不要。Web Audio APIで4波形の試聴再生を忠実に再現可能 |
| 実装方式 | 方式B（ZIP+TOMLを自前生成） | GUIの自由度が要件（使いやすい打ち込みUI）に直結。フォーマットは`format_version = 1`のテキスト形式で実装難度が低い |
| MVP範囲 | ピアノロール編集＋pyxres書き出し＋試聴再生 | 後述§9のフェーズ計画参照。pyxres読み込み・パターン共有UIはフェーズ2 |
| 外部依存 | ゼロ（ライブラリ不使用） | ZIPは無圧縮（STORED）エントリなら約100行で自前実装可能。TOML生成は文字列テンプレートで足りる。単一ファイル配布と長期保守性を優先 |

## 2. アーキテクチャ

単一HTMLファイル内を5モジュールに分離する。依存方向は一方向（UI→Model、AudioEngine/Exporter→Model）とし、ModelはDOM・Web Audioに依存しない純粋データ層とする。これはエクスポート処理とUIのテスト・変更を独立させるための標準的な分離である。

```
┌─────────────────────────────────────────┐
│  UI層                                    │
│  PianoRollView / SongListView /          │
│  TransportBar / SoundPropertyPanel       │
├──────────────┬──────────────────────────┤
│  AudioEngine │  Exporter                 │
│  (Web Audio) │  (TOML生成 + ZIP梱包)      │
├──────────────┴──────────────────────────┤
│  Model（プロジェクトデータ・バリデーション）  │
├─────────────────────────────────────────┤
│  Storage（localStorage自動保存 / JSON入出力）│
└─────────────────────────────────────────┘
```

## 3. データモデル

### 3.1 内部モデルとエクスポートモデルの分離

実現可能性調査§4の帰結に従い、エディタ内部のデータモデルとpyxresの書き出しモデルを分離する。内部モデルは上限を持たず、書き出し時に検証・割り当てを行う。

```
内部モデル（JSON）                     pyxres（書き出し時に生成）
Project                               pyxel_resource.toml
 ├─ patterns: Pattern[]   ──選択──▶   [[sounds]] × 最大64
 └─ songs: Song[]         ──選択──▶   [[musics]] × 最大8
```

### 3.2 スキーマ

```jsonc
{
  "formatVersion": 1,          // 内部フォーマットのバージョン（pyxresのversionとは別物）
  "meta": { "title": "", "created": "", "modified": "" },
  "patterns": [                // フレーズ（pyxresのSoundに対応）
    {
      "id": "p1",              // 内部参照用ID。曲間・曲内で共有可能
      "name": "イントロベース",  // 内部のみ。pyxresには保存されない
      "notes": [24, -1, 26],   // -1=休符, 0〜59。pyxres仕様と同一表現
      "tones": [1],            // 循環配列。pyxres仕様と同一表現
      "volumes": [7],
      "effects": [0],
      "speed": 30              // 既定値30（pyxel-core settings.rs の DEFAULT_SOUND_SPEED）
    }
  ],
  "songs": [                   // 曲（pyxresのMusicに対応）
    {
      "id": "s1",
      "name": "メインテーマ",
      "channels": [            // 最大4チャンネル
        ["p1", "p2", "p1"],    // パターンIDの並び。IDによる参照＝パターン共有の実体
        ["p3"]
      ]
    }
  ],
  "export": {                  // 書き出し割り当て（内部モデル→pyxres 8枠のマッピング）
    "musicSlots": ["s1", null, "s2", ...]  // 長さ8。nullは空トラック
  }
}
```

設計上の要点は3つ。第一に、チャンネル列がパターンを**IDで参照**する構造そのものがパターン共有であり、追加機能ではなくモデルの基礎とする（調査書§4「64枠制約下の必須設計」の実装）。フェーズ2で追加するのは共有状況の可視化UIのみで、データ構造の変更は不要。第二に、`notes`/`tones`等の値表現をpyxres仕様と同一にすることで、書き出し時の変換を「参照解決と割り当て」だけに限定し、変換バグの混入面を最小化する。第三に、内部モデルは曲数・パターン数無制限とし、8曲超の管理は`export.musicSlots`での取捨選択で吸収する（調査書§4後段の運用方針）。

### 3.3 書き出し時の割り当てアルゴリズム

1. `export.musicSlots`で選択された曲が参照するパターンIDを収集し、重複排除する（Set化）。
2. ユニークパターン数が64を超えたら書き出しを拒否し、超過数と各曲の消費パターン数を提示する。
3. パターンIDを`sounds[0..n]`へ登場順に割り当て、対応表（ID→index）を作る。
4. 各曲の`channels`をindex列に変換し`musics[slot].seqs`とする。5チャンネル以上は入力段階で禁止。
5. 未使用のsoundsスロットは空エントリ（`notes = []`等）で埋める。※Pyxelは枠数未満のファイルも受理するが、通常セーブと同じ64エントリ構成に揃えて差異を作らない。

## 4. UI設計

### 4.1 画面構成

```
┌────────────────────────────────────────────────┐
│ TransportBar: ▶再生 ■停止 | speed | 書き出し     │
├──────────┬─────────────────────────────────────┤
│ SongList │ ChannelLane: ch0 [p1][p2][p1]  ← 曲構造│
│ Pattern  │              ch1 [p3]                │
│ List     ├─────────────────────────────────────┤
│          │ PianoRoll: 選択中パターンの編集         │
│          │  縦60行(C0〜B4) × 横 note数            │
│          ├─────────────────────────────────────┤
│          │ PropertyPanel: tone/volume/effect/speed│
└──────────┴─────────────────────────────────────┘
```

### 4.2 ピアノロール仕様

- **グリッド**: 縦60行固定（note値0〜59、C0〜B4）。pyxresの音域上限をUIの物理的制約として表現し、範囲外入力を構造的に不可能にする。横軸は1列=1ノート（`notes`配列の1要素）。
- **入力**: クリックで配置/削除。同一列に複数音は置けない（Pyxelの1サウンド=単音仕様のため、1列1音をUIで強制）。ドラッグで移調・移動。
- **休符**: 音のない列は自動的に`-1`。明示的な休符操作を不要にする。
- **ノート単位のプロパティ**: `tones`/`volumes`/`effects`は循環配列のため、UIでは「全体一括」と「ノート個別」の2モードを持つ。個別編集した時点で配列を`notes`と同長に展開する（循環仕様との互換は保たれる）。
- **描画**: `<canvas>` 1枚。60×百数十セル程度の再描画は全面クリア→全描画で十分（16ms予算に対し余裕）。仮想化等の最適化は不要と判断。

### 4.3 チャンネルレーン仕様

曲構造（`seqs`）の編集ビュー。パターンをブロックとして横に並べ、ドラッグ＆ドロップで順序変更、既存パターンのドロップで再利用配置を行う。ブロック幅はパターンの実再生時間（`notes.length × speed / 120`秒）に比例させ、チャンネル間の時間軸を視覚的に揃える。

## 5. 音声エンジン設計（試聴再生）

Pyxelの実装（pyxel-core、後述§10の一次情報）に準拠した再現仕様。「エディタで聴いた音とPyxel実機の音が一致する」ことを品質基準とする。

### 5.1 再現パラメータ（pyxel-coreソースより確定）

| 項目 | 値 | 出典（pyxel-core） |
| --- | --- | --- |
| サンプルレート | 22,050 Hz | settings.rs `AUDIO_SAMPLE_RATE` |
| 周波数変換 | MIDI note = pyxres note + 36（ノイズ音のみ +60）、f = 440×2^((midi−69)/12) | sound.rs `base_note`（Wavetable=36 / Noise=60）、voice.rs `A4_FREQUENCY` |
| 音量 | 線形。level = volume / 7 | sound.rs `volume as f32 / MAX_VOLUME` |
| tick | 1/120秒。speed = 1ノートのtick数 | settings.rs `SOUND_TICKS_PER_SECOND` |
| 三角波 | 32ステップ・4bitウェーブテーブル、ゲイン1.0 | settings.rs `DEFAULT_TONE_TRIANGLE` |
| 矩形波 | デューティ50%（[1,0]）、ゲイン0.3 | settings.rs `DEFAULT_TONE_SQUARE` |
| パルス波 | デューティ25%（[1,0,0,0]）、ゲイン0.3 | settings.rs `DEFAULT_TONE_PULSE` |
| ノイズ | 長周期LFSR（NES APU方式、tap bit 1・シード0x7001・周期32767サンプル）、ゲイン0.6 | settings.rs `DEFAULT_TONE_NOISE`、voice.rs `set_noise` |
| チャンネルゲイン | 0.125 | settings.rs `DEFAULT_CHANNEL_GAIN` |
| Slide (1) | 直前ノートから現ノートへ、ノート長かけてピッチ補間 | channel.rs `last_midi_note` |
| Vibrato (2) | 三角LFO、周期20tick（6Hz）、深さ±25セント、遅延なし | settings.rs `VIBRATO_PERIOD_TICKS` / `VIBRATO_DEPTH_CENTS` |
| FadeOut (3) | 音量1.0→0.0をspeed全tickで線形フェード | sound.rs `EnvelopeSet` |
| Half (4) | 前半保持、後半speed/2 tickでフェード | 同上 |
| Quarter (5) | 前3/4保持、後speed/4 tickでフェード | 同上 |

### 5.2 実装方式

`AudioWorklet`ではなく`AudioBufferSourceNode`方式を採用する。すなわち、再生要求時にパターン/曲全体をJSでオフラインレンダリングして22,050HzのPCMバッファを生成し、一括再生する。根拠：(1) エディタの試聴はレイテンシ要件が緩く、リアルタイム合成が不要。(2) レンダラは純関数になり、Pyxel実機出力とのWAV比較テストがそのまま書ける。(3) 同一レンダラをWAV書き出し機能（フェーズ3候補）に転用できる。ループ再生は`AudioBufferSourceNode.loop`で対応。編集中の単音プレビュー（クリック時に鳴らす）のみ、短いバッファの即時生成で行う。

### 5.3 忠実度の割り切り

Pyxel実機はNES APUクロック（1,789,773Hz）単位で発振位相を刻み、ノート境界に約1msのゲイン補間を入れている（voice.rs `NOTE_INTERP_CLOCKS`）。MVPではクリックノイズ防止の1ms線形クロスフェードのみ再現し、クロック粒度の位相計算はサンプル粒度で近似する。可聴差は実用上無視できる範囲と見込むが、フェーズ2でPyxel実機のWAV出力との差分検証を行い、必要なら精度を上げる。

## 6. pyxres書き出し仕様

### 6.1 TOML生成

文字列テンプレートで直接生成する（汎用TOMLライブラリ不要。出力するのは数値配列とu16スカラーのみで、エスケープ問題が発生しない）。公式仕様に従い、`format_version = 1`と4セクションすべてを出力する。音楽エディタのため画像・タイルマップは空配列とする：

```toml
format_version = 1

images = []
tilemaps = []

[[sounds]]
notes = [24, 26, 28, -1]
tones = [1]
volumes = [6]
effects = [0]
speed = 20
# … 64エントリ

[[musics]]
seqs = [[0, 1], [2]]
# … 8エントリ
```

空セクションを`images = []`形式で書くのは、Pyxel自身が`save(..., exclude_images=True)`で出力する形式と同一であり、公式仕様が「空配列として書かれる」と明記しているため（pyxres-format.md）。musicsの空トラックは`seqs = []`、末尾の空チャンネルは省略（同仕様）。

### 6.2 ZIP梱包

無圧縮（method 0 = STORED）のZIPを自前実装する。必要なのはローカルファイルヘッダ＋セントラルディレクトリ＋終端レコードの生成とCRC-32のみ（CRC-32はテーブル方式で約20行）。エントリは`pyxel_resource.toml`の1つだけであり、TOMLは高々数十KBのため圧縮の実益がない。ZIP仕様上、STOREDは全ZIPリーダーの必須サポート項目であり互換リスクはない。ダウンロードは`Blob`＋`<a download>`で行う。

### 6.3 書き出し前バリデーション

| 検査 | 条件 | 失敗時 |
| --- | --- | --- |
| パターン数 | ユニーク参照パターン ≤ 64 | 超過数を提示しブロック |
| 曲数 | 割り当てスロット ≤ 8 | 割り当てUIで構造的に超過不可 |
| チャンネル数 | ≤ 4 | 入力段階で構造的に超過不可 |
| note範囲 | -1〜59 | ピアノロールで構造的に超過不可 |
| volume/effect/speed | 0〜7 / 0〜5 / 1〜65535 | 入力段階で制限 |
| 空曲 | seqsが全チャンネル空 | 警告のみ（保存は許可。Pyxel側も空を許容） |

「構造的に不可能」を基本戦略とし、事後エラーは64枠超過のみに限定する。これは制約の大半が固定範囲であり、UIの選択肢自体を範囲内に閉じられるため。

## 7. 永続化

- **自動保存**: 編集操作のたびに内部モデルJSONをlocalStorageへ保存（debounce 1秒）。ブラウザ単体構成での不慮のデータ消失に対する最低限の防御。
- **明示保存**: 内部モデルを`.json`としてダウンロード／ファイル選択で読み込み。localStorageはドメイン・ブラウザに束縛されるため、可搬性はJSONファイルで担保する。
- pyxresは「書き出し専用フォーマット」と位置づけ、往復編集の正本は常に内部JSONとする。pyxresには`name`等の内部メタデータが保存できないため、pyxresを正本にすると情報が不可逆に落ちる。

## 8. 制約の明示（UX要件）

調査書§6の通り、`.pyxres`の表現力上限（60音階・4波形・8段階音量・6種エフェクト、MML表現は保存不可）はフォーマット固有の制約である。エディタはこの上限を超える編集機能を持たない設計とし（超えられるのに書き出せない、という不整合を作らない）、初回起動時のヘルプにこの制約を明記する。

## 9. 開発フェーズ

| フェーズ | 内容 | 完了基準 |
| --- | --- | --- |
| 1 (MVP) | ピアノロール編集・チャンネルレーン・試聴再生・pyxres書き出し・localStorage/JSON永続化 | 本エディタで作った2曲入りpyxresがPyxel 2.9.x実機の`pyxel.load()`＋`pyxel.playm()`で正常再生される |
| 2 | pyxres読み込み（画像・タイルマップの保全含む）、パターン共有の可視化UI、実機WAVとの音質差分検証 | 実機生成のpyxresを読み込み→無編集で再書き出しした際、sounds/musicsが完全一致する |
| 3 (候補) | WAV書き出し、MIDIインポート、undo/redo拡充 | — |

フェーズ1の完了基準を「実機での再生成功」に置くのは、本プロジェクトの価値がフォーマット互換性そのものにあるため。開発初期からPyxel実機（ヘッドレスモード）での読み込みテストをCI的に回す。

## 10. リスク

| リスク | 評価 | 対応 |
| --- | --- | --- |
| format_versionの将来変更 | 低（公式が後方互換方針を明記、読み込み側はv4まで許容） | 書き出しはv1固定を維持。Pyxelリリースノートの監視 |
| 音質再現の不一致 | 中（クロック粒度の近似による） | フェーズ2で実機WAV差分検証。レンダラが純関数のため検証容易 |
| 既存ツール（Pyxel Composer）との競合 | 低 | 「1プロジェクト複数曲＋pyxres直接書き出し」は当該ツールで未確認の差別化点（調査書§7） |
